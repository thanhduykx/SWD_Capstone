# CPMS Swagger Testing Guide

Use this guide when FE needs to test APIs manually through Swagger.

## Start Here

1. Run backend and open `http://localhost:5122/swagger`.
2. Open `00 - FE Test Guide`.
3. Call `GET /api/test-support/swagger-guide`.
4. Call `POST /api/auth/login`.
5. Copy `accessToken`.
6. Click `Authorize` in Swagger.
7. Paste the token value only.
8. Call protected endpoints.

The `api/test-support/*` endpoints are intended for local testing. They run in `Development` by default. Docker local testing enables them with `Swagger__ExposeTestSupport=true`; do not enable that flag for a real production deployment.

## Local Test Accounts

These accounts are for local development only.

| Role | Username | Password |
| --- | --- | --- |
| TrainingDepartment | `test.training` | `Test@123456` |
| Lecturer | `test.lecturer` | `Test@123456` |
| Lecturer | `test.lecturer.nodept` | `Test@123456` |
| Student | `test.student` | `Test@123456` |

Moderator is the main operator for account management. `SystemAdministrator` is kept only for compatibility with old local data and is not part of the normal FE test flow.

If one account is missing in a fresh database, login as moderator and create it through `POST /api/accounts`.

## Rules FE Should Not Guess

- Use GET APIs first to retrieve ids.
- `DateOnly` uses `yyyy-MM-dd`.
- `DateTime` uses ISO format, for example `2026-06-15T00:00:00Z`.
- Review `dayOfWeek` uses Monday=1 through Sunday=7.
- Review and defense `slot` must be 1 through 8.
- Account username is generated by backend from `fullName` and `identityCode`.
- Example: `Dương Thành Thanh Duy` + `SE194673` becomes `DuyDTTSE194673`.
- Lecturer account creation does not require `department`.
- TrainingDepartment account creation still uses `department` as the office/department name.

## Recommended Test Flows

### Account Flow

1. Login as `test.training`.
2. Authorize Swagger.
3. Call `GET /api/accounts`.
4. Call `POST /api/accounts`.
5. Login as the new account to verify credentials.

### Lecturer Availability Flow

1. Login as `test.lecturer`.
2. Authorize Swagger.
3. Call `GET /api/semesters/resolve?date=2026-06-15`.
4. Call `PUT /api/review-availability/week`.
5. Call `GET /api/review-availability/week`.

### Review Scheduling Flow

1. Login as `test.training`.
2. Authorize Swagger.
3. Call `GET /api/review-scheduling/board`.
4. Use returned `lecturerId` and `groupId`.
5. Call `POST /api/review-sessions/bulk-assign`.
6. Call `POST /api/review-schedules/publish`.

### Review Checklist Flow

1. Login as assigned lecturer.
2. Call `GET /api/review-sessions/my`.
3. Use returned `submissionId`.
4. Call `GET /api/review-submissions/{submissionId}`.
5. Save with `PUT /api/review-submissions/{submissionId}/draft`.
6. Submit with `POST /api/review-submissions/{submissionId}/submit`.

### Defense Flow

1. Login as `test.training`.
2. Create defense round.
3. Create defense board.
4. Assign project to board.
5. Login as an assigned lecturer.
6. Resolve defense session.
7. Chairman starts scoring.
8. Council members submit scores.
9. Chairman closes scoring.
